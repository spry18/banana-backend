Complete Changes Summary — Extra Vehicle & Extra Day Feature
What Was Done
Two frontend improvement requests were implemented:

Extra Vehicle (Overflow) — When a vehicle is full, OM/Munshi can add an extra vehicle with full assignment details
Extra Day (Rollover) API Audit — Verified existing APIs & improved OM Plots to return all logistics records

Works alongside existing isRollover and parentAssignmentId fields. Overflow = extra vehicle, Rollover = extra day.

2. 
logistics.controller.js
Change: Added 2 new controller functions

addExtraVehicle — Creates overflow assignment
Route: POST /api/logistics/:id/add-vehicle
Access: Admin, Operational Manager, Munshi
Required field: driverId
Optional fields (fallback to original if not sent):
Field	Type	Example
munshiId	ObjectId	Different munshi for extra vehicle
companyId	ObjectId	Different company
purchaseRate	Number	918
totalBoxes	Number	200
priority	String	HIGH / MEDIUM / LOW
lightInTime	String	"05:00 AM"
lightOutTime	String	"07:00 AM"
teamName	String	"Overflow Crew-01"
scheduledDate	Date	"2026-04-30"
Creates: New Logistics record with isOverflow: true, parentAssignmentId → original

Sends 3 WhatsApp notifications:

Extra driver → "You have been assigned as extra vehicle..."
Munshi (could be different) → "An extra vehicle driven by X added..."
Original driver → "An extra vehicle added for support..."
Validations:

Original assignment must exist and NOT be COMPLETED/CANCELLED/APPROVED
Driver must exist and have a vehicle linked
Extra driver cannot be the same as original driver
If munshiId provided, validates munshi exists
getRelatedAssignments — Returns full assignment chain
Route: GET /api/logistics/:id/related
Access: Admin, Operational Manager, Munshi
Response:
json
{
    "original": { ... },
    "rollovers": [ ... ],
    "overflows": [ ... ],
    "totalRelated": 3
}
Walks up parentAssignmentId chain to find root, then fetches all children + grandchildren
